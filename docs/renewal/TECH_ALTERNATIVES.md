# 기술 대체 방안 비교

> 리뉴얼 과정에서 검토한 기술 선택지와 채택/비채택 근거를 기록한다.
> 구현 진행 중 변경 사항이 발생하면 이 문서를 업데이트한다.

---

## 1. 빌드 도구

### 검토 대상

| 도구 | 장점 | 단점 | 판정 |
|------|------|------|------|
| **Vite 6** | ESBuild 기반 초고속 HMR, 활발한 생태계, React/TS 기본 지원 | CRA에서 마이그레이션 필요 | **채택** |
| Next.js 15 | SSR/SSG, 파일 기반 라우팅, 이미지 최적화 | 관리자 대시보드에 SSR 불필요, 학습 비용 높음, 기존 코드 전면 재작성 필요 | 비채택 |
| Turbopack | Vercel 개발, Rust 기반 | Next.js 전용, 독립 사용 불가 | 비채택 |
| Rspack | Webpack 호환, Rust 기반 고속 | 아직 생태계 초기, 플러그인 부족 | 대기 |

### 채택 근거
- 관리자 대시보드는 SPA로 충분 (SSR 불필요)
- CRA → Vite 전환 비용이 가장 낮음 (소스 코드 변경 최소)
- Docker 개발 환경에서 HMR 성능 차이가 체감됨

## 2. 상태관리 (클라이언트)

### 검토 대상

| 라이브러리 | 번들 크기 | 보일러플레이트 | 학습 곡선 | 판정 |
|-----------|----------|--------------|----------|------|
| **Zustand** | ~1KB | 최소 | 낮음 | **채택** |
| Jotai | ~2KB | 최소 | 낮음 | 대안 |
| Redux Toolkit | ~11KB | 중간 | 중간 | 비채택 |
| MobX (현재) | ~16KB | 높음 (데코레이터) | 중간 | 제거 |
| Recoil | ~20KB | 중간 | 중간 | 비채택 (Meta 지원 축소) |

### 채택 근거 (Zustand)
- 현재 프로젝트의 클라이언트 상태가 단순 (UI 토글, 인증 정보)
- React Query가 서버 상태를 이미 관리 → 클라이언트 상태 관리는 경량이면 충분
- Provider 래핑 불필요, 컴포넌트 외부에서도 접근 가능
- TypeScript 친화적

### Jotai를 대안으로 남긴 이유
- 원자(atom) 기반으로 세밀한 리렌더링 제어 가능
- 향후 복잡한 상태가 추가되면 전환 검토

## 3. HTTP 클라이언트

### 검토 대상

| 라이브러리 | 장점 | 단점 | 판정 |
|-----------|------|------|------|
| **Axios** | 인터셉터, 토큰 갱신 로직 구현 완료 | 번들 크기 다소 큼 | **채택** (유지) |
| ky | 경량, fetch 기반 | 기존 인터셉터 재작성 필요 | 비채택 |
| ofetch (unjs) | 경량, 자동 JSON, 리트라이 | 생태계 초기 | 대기 |
| Superagent (현재) | - | 체이닝 API 복잡, 인터셉터 미지원 | **제거** |

### 채택 근거 (Axios 유지)
- `src/utils/request.js`에 토큰 갱신 큐, 디바이스 감지 등 복잡한 인터셉터가 이미 구현됨
- 이를 다른 라이브러리로 재작성하는 비용 대비 이점 없음
- Superagent만 제거하고 Axios로 단일화

## 4. UI 컴포넌트 라이브러리

### 검토 대상

| 라이브러리 | 장점 | 단점 | 판정 |
|-----------|------|------|------|
| **MUI 6** | 기존 MUI 5 기반, 마이그레이션 비용 낮음, CSS 변수 테마 | 번들 크기 | **채택** |
| Ant Design 5 | 관리자 UI에 최적화된 컴포넌트 다수 | 기존 MUI 코드 전면 교체 필요, 이미 antd 5 부분 설치됨 | 비채택 |
| Chakra UI | 접근성 우수, 간결한 API | MUI에서 전면 전환 비용 과대 | 비채택 |
| shadcn/ui | 커스텀 자유도 최고, Tailwind 기반 | 스타일 시스템 전면 교체 필요 (Emotion → Tailwind) | 장기 검토 |

### 채택 근거 (MUI 6)
- 기존 코드가 MUI 5 기반 → MUI 6 업그레이드가 가장 효율적
- MD* 래퍼 컴포넌트를 제거하고 MUI 6 직접 사용으로 전환
- CSS 변수 기반 테마로 다크모드 전환 성능 개선

### Ant Design 참고 사항
- `antd 5.27.3`이 이미 설치되어 있으나 활용도 불명확
- MUI 6 전환 완료 후 antd 의존성 제거 검토

## 5. 테이블 컴포넌트

### 검토 대상

| 라이브러리 | 서버사이드 | 가상화 | TypeScript | 판정 |
|-----------|----------|--------|-----------|------|
| **TanStack Table 8** | 지원 | 지원 | 네이티브 | **채택** |
| AG Grid | 지원 | 지원 | 지원 | 비채택 (라이선스 비용) |
| MUI DataGrid | 지원 | 지원 | 지원 | 대안 |
| react-table 7 (현재) | 지원 | 제한적 | @types 필요 | 교체 |

### 채택 근거 (TanStack Table 8)
- react-table 7의 공식 후속 버전 (동일 개발자)
- 헤드리스 UI → MUI 컴포넌트와 자유롭게 조합 가능
- 백엔드 20개+ 목록 API가 모두 페이지네이션 지원 → 서버사이드 페이지네이션 활용

### MUI DataGrid를 대안으로 남긴 이유
- MUI 생태계 통합이 자연스러움
- 단, 무료 버전(Community)의 기능 제한 있음 (컬럼 그룹, 트리 등)

## 6. 폼 관리

### 검토 대상

| 라이브러리 | 리렌더링 | 검증 통합 | TypeScript | 판정 |
|-----------|---------|----------|-----------|------|
| **React Hook Form** | 최소 (비제어) | Zod/Yup 어댑터 | 네이티브 | **채택** |
| Formik | 전체 (제어) | Yup 기본 | @types 필요 | 비채택 |
| React Final Form | 최소 | 커스텀 | 제한적 | 비채택 |

### 검증 라이브러리

| 라이브러리 | TypeScript 추론 | 번들 크기 | 판정 |
|-----------|----------------|----------|------|
| **Zod** | 네이티브 (infer) | ~13KB | **채택** |
| Yup (현재) | @types 필요 | ~15KB | 교체 |
| Valibot | 트리셰이킹 최적 | ~1KB | 대안 |

### 채택 근거
- React Hook Form: 20개+ CRUD 폼에서 리렌더링 최소화 효과 큼
- Zod: `z.infer<typeof schema>`로 폼 타입을 스키마에서 자동 추출 → 타입 중복 제거

## 7. 리치 텍스트 에디터

### 검토 대상

| 라이브러리 | 유지보수 상태 | 확장성 | 판정 |
|-----------|-------------|--------|------|
| **TipTap** | 활발 | 플러그인 아키텍처 | **채택** |
| Plate | 활발 | shadcn 기반 | 대안 |
| react-quill (현재) | 유지보수 중단 | 제한적 | 교체 |
| CKEditor 5 | 활발 | 높음 | 비채택 (라이선스) |
| Lexical (Meta) | 활발 | 높음 | 대안 |

### 채택 근거 (TipTap)
- ProseMirror 기반으로 안정적
- 마크다운, 이미지 업로드, 테이블 등 관리자에 필요한 기능 플러그인으로 제공
- React 통합 공식 지원

### Lexical을 대안으로 남긴 이유
- Meta(Facebook) 개발로 장기 지원 기대
- 저수준 API라 커스텀 자유도 높지만, 초기 구현 비용도 높음

## 8. 차트 라이브러리

### 검토 대상

| 라이브러리 | 선언적 API | React 통합 | 판정 |
|-----------|----------|-----------|------|
| **Chart.js + react-chartjs-2** (현재) | 중간 | 래퍼 | **유지** |
| Recharts | 높음 (JSX) | 네이티브 | 대안 |
| Nivo | 높음 | 네이티브 | 대안 |
| ECharts (Apache) | 중간 | 래퍼 | 비채택 (과도) |

### 유지 근거 (Chart.js)
- 대시보드 차트가 현재 안정적으로 동작
- 10종 차트 컴포넌트(`examples/Charts/`)가 이미 구현됨
- 교체 비용 대비 이점 불명확

### 향후 교체 조건
- 실시간 대시보드 요구 발생 시
- 복잡한 인터랙티브 차트 필요 시
- → 이 경우 Recharts 또는 Nivo 검토
